# PixelTravelMap 三步创建器可用性测试手册

版本：2026-07-22

## 1. 本轮要回答的问题

这轮不是测试用户是否喜欢像素风，而是判断三步创建流程是否足够简单：

1. 用户能否把已有行程导入并看到结构化草稿。
2. 用户能否发现并修正一个解析错误。
3. 用户能否生成地图，并找到第二天的同行简报。

本轮不验证市场规模、付费意愿、长期留存或真实旅行中的重规划价值。

## 2. 参与者

- 5 名最近一年内规划过两天以上自由行的人。
- 至少 3 名曾经负责整理多人行程。
- 不要求熟悉地图 API、经纬度或开发工具。
- 不把项目作者本人计入 5 名参与者。

## 3. 测试材料

- 测试地址：`https://leoxin99.github.io/PixelTravelMap/dist/builder.html`
- 一份参与者自己使用过的非敏感行程，或统一测试行程。
- 屏幕录制或观察记录，仅在参与者明确同意后使用。
- 每位参与者独立完成，避免看到其他人的操作和答案。

## 4. 主持开场

> 今天测试的是产品，不是测试你。页面中任何不清楚的地方都算产品需要改进。请按你平时的习惯操作，并尽量说出你正在寻找什么。过程中我不会教你怎么做；卡住时可以直接告诉我。

不要提前解释“城市锚点”“Poster”“Artifact”或操作按钮的位置。

## 5. 三项任务

### 任务一：导入已有行程

> 这是你准备发给同行朋友的一份行程。请把它放进页面，并让产品整理出来。当你认为第一步完成时告诉我。

观察首次点击目标、是否误以为必须填写经纬度，以及从开始到看到结构化草稿的时间。

### 任务二：修正一个错误

提前在测试文本中放入一个明显错误，例如把“中央大街”写到错误城市，或漏掉一个地点。

> 你发现行程里有一处不对，请把它改正确，并继续生成旅行地图。

观察能否找到对应 Day 和地点、是否理解“城市方位待补全”、是否逐景点重复定位，以及修改后旧结果是否正确失效。

### 任务三：找到明日简报

> 现在你准备把第二天的安排发给同行朋友。请找到适合发送的内容，并选择第二天。

观察能否找到并切换每日简报、是否理解互动地图与每日简报的区别，以及是否把故事地图误认为精确导航。

## 6. 测后问题

1. 哪一步最费力？为什么？
2. 页面里哪些内容你认为可以删掉？
3. 你会在旅行前的什么时刻使用每日简报？
4. 如果行程临时改变，你最希望它先帮你处理什么？
5. 你会把哪一种结果发给同行者？为什么？

避免询问“你是不是觉得这个功能很方便”之类的诱导问题。

## 7. 单场记录模板

| 字段 | 记录 |
| --- | --- |
| 参与者编号 | P01-P05 |
| 旅行规划经验 | 近期行程类型、是否负责多人组织 |
| 设备 | 手机 / 电脑 |
| 任务一 | 完成 / 需提示 / 未完成；用时；关键卡点 |
| 任务二 | 完成 / 需提示 / 未完成；用时；关键卡点 |
| 任务三 | 完成 / 需提示 / 未完成；用时；关键卡点 |
| 首次误点 | 按钮或区域 |
| 原话 | 仅记录真实参与者原话 |
| 观察事实 | 实际点击、停顿、返回、求助 |
| 主持人推断 | 与观察事实分开填写 |
| 严重度 | 阻断 / 明显摩擦 / 轻微问题 |

## 8. 预注册决策 Gate

- 至少 4/5 人无需主持人指导完成三项任务。
- 至少 4/5 人在 3 分钟内看到第一张地图。
- 主路径中没有参与者被要求手动输入经纬度。
- 至少 4/5 人能找到并选择正确的每日简报。
- 没有人把故事地图误认为精确导航工具。

如果任务一失败较多，优先修改输入与解析；任务二失败较多，优先修改纠错和城市方位；任务三失败较多，优先调整结果页信息层级。不要用视觉润色掩盖流程失败。

## 9. 报告边界

- 5 人测试用于发现可用性问题，不用于推断市场比例。
- 合成 Agent 的回答不得混入真人原话。
- 未完成的任务不得写成“用户已验证”。
- 结果报告同时保留成功、失败、分歧和反例。
